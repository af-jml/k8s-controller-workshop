What is Kubernetes, really?

A declarative system built on control loops.

You declare

kind: Deployment
spec:
  replicas: 3

"I want 3 of these."

Kubernetes reconciles

Controllers continuously compare desired vs actual and act to close the gap.

0 running? Create 3. One dies? Make another.

The reconciliation loop

for {
  desired := readSpec()        // what the user asked for
  actual  := observeWorld()    // what currently exists
  if actual != desired {
    act(desired, actual)       // create / update / delete
  }
}

This single idea explains most of Kubernetes.

What's inside the cluster?

Control plane

  • kube-apiserver ‚Äî the front door; the only thing that talks to etcd
  • etcd ‚Äî the database; all cluster state lives here
  • scheduler ‚Äî assigns pods to nodes
  • controller-manager ‚Äî runs the built-in control loops

On every node

  • kubelet ‚Äî starts & supervises containers
  • kube-proxy ‚Äî wires up Service networking

The API server is the hub

  • Everything is a REST resource at /apis/<group>/<version>/...
  • Components don't call each other ‚Äî they watch the API.
  • A watch is a long-lived stream of add/update/delete events.
kubectl get pods --watch
# the same mechanism every controller uses

How a pod "responds" to a resource

  1. A process in a pod opens a watch on some resource type.
  2. The API server streams every change to that process.
  3. The process reacts ‚Äî creating, updating, deleting other resources.
  4. It writes results back to the API (e.g. a status).

That process is a controller. That's the whole trick.

Extending the API: CRDs

A CustomResourceDefinition teaches the API server a brand new resource type ‚Äî with its own schema, validation, and kubectl support.

apiVersion: apiextensions.k8s.io/v1
kind: CustomResourceDefinition
metadata:
  name: reportrequests.reports.workshop.io
spec:
  group: reports.workshop.io
  names: { kind: ReportRequest, plural: reportrequests }
  scope: Namespaced
  versions: [ ... schema ... ]

A CRD alone does nothing

After applying a CRD you can do this‚Ä¶

kubectl apply -f my-reportrequest.yaml
kubectl get reportrequests

You will feel this in step 03.

The operator / controller pattern

Pair a CRD (the data) with a controller (the behaviour):

CRD            ‚Üí  a new noun in the API
Controller     ‚Üí  the verb that makes it real
Together       ‚Üí  an "Operator"

Encodes the knowledge a human operator would otherwise apply by hand.

Real-world use cases

  • Databases that provision & back themselves up (CloudNativePG, Zalando Postgres)
  • Certificate management (cert-manager: Certificate CRDs)
  • GitOps (Argo CD: Application CRDs)
  • Cloud resources from inside the cluster (Crossplane)
  • Your own domain objects ‚Äî like today's ReportRequest.

Why this design teaches well

  • The controller creates other Kubernetes objects (Jobs) ‚Äî the real operator pattern, visible in k9s.
  • Owner references mean deleting a request garbage-collects its Job.
  • The status subresource shows desired-vs-actual in action.
  • A few seconds of "work" make Pending ‚Üí Processing ‚Üí Completed visible.
