Two shifts driving adoption

Beyond "containers are nice", two forces are pushing Kubernetes to the centre right now:

🇪🇺 Sovereign clouds

Who controls — and can legally access — your data and infrastructure?

🤖 AI hosting

Where do you run GPUs, models, and long batch jobs?

Both end up needing the same thing: a portable, automatable control plane.

These are the two conversations clients are actually having in 2025/26. Kubernetes is the common denominator under both — that's why it's worth understanding deeply.

Force 1: Digital sovereignty

  • Regulation & risk: GDPR, Schrems II and the US CLOUD Act make data location and control a legal question, not just a technical one.
  • Reducing dependency on the three US hyperscalers — see Gaia-X and sovereign offers like StackIT, OVHcloud, IONOS, Open Telekom Cloud, T-Systems.
  • Kubernetes is the portability layer: the same declarative API on any provider or on-prem — so you can move, repatriate, or multi-cloud without a rewrite.
Sovereignty isn't only "EU data centre" — it's not being locked in to one vendor's APIs.
The key insight: a managed proprietary cloud service ties you to that vendor's API. A Kubernetes workload is described in vendor-neutral YAML and runs anywhere — that optionality is itself a form of sovereignty.

Force 2: Hosting AI yourself

  • Why self-host: data privacy, cost control at scale, latency, and avoiding per-token lock-in to a single model API.
  • GPUs are first-class on K8s: the NVIDIA GPU Operator, device plugins, MIG / time-slicing, and Dynamic Resource Allocation schedule scarce accelerators.
  • An ecosystem already lives here: KServe, Kubeflow, Ray, and vLLM all run on Kubernetes — it's becoming the default substrate for ML platforms.
Inference, training, fine-tuning, evals, batch backfills
  → long-running, stateful, retry-heavy, GPU-hungry
  → exactly the workloads controllers are good at
Note the overlap with sovereignty: many teams self-host models precisely because they can't send sensitive data to a third-party API. The two forces reinforce each other.

The common thread

Both stories converge on Kubernetes for the same two reasons:

Portability   →  vendor-neutral API; run the same thing anywhere
Automation    →  control loops manage scale, retries, policy, cost

…and automation is exactly what controllers are — which is what we build today.

This is the hand-off into the rest of the deck: portability gets you onto Kubernetes, but the value comes from automation, and automation is controllers.

Why controllers in the AI era?

  • AI workflows are long-running and stateful: queue, process, retry, publish.
  • Controllers encode operational policy as code (who acts, when, and how).
  • CRDs let product teams expose domain APIs without leaking infrastructure details.

What is Kubernetes, really?

A declarative system built on control loops.

You declare

kind: Deployment
spec:
  replicas: 3

"I want 3 of these."

Kubernetes reconciles

Controllers continuously compare desired vs actual and act to close the gap.

0 running? Create 3. One dies? Make another.

The reconciliation loop

for {
  desired := readSpec()        // what the user asked for
  actual  := observeWorld()    // what currently exists
  if actual != desired {
    act(desired, actual)       // create / update / delete
  }
}

This single idea explains most of Kubernetes.

Core resource types

ResourceWhat it is
PodOne or more containers, scheduled together
ReplicaSetKeeps N copies of a pod running
DeploymentManages ReplicaSets; rollouts & rollbacks
ServiceStable network endpoint for a set of pods
IngressHTTP routing from outside into Services
ConfigMap / SecretConfiguration & sensitive data
NamespaceA scope to group and isolate resources
JobRun a pod to completion (we'll use these!)

They build on each other

Deployment
   └── ReplicaSet
          └── Pod
                 └── Container(s)

Service  ──routes-to──▶  Pods
Ingress  ──routes-to──▶  Service

Each layer is itself reconciled by a controller.

Appendix: what's inside the cluster?

Control plane

  • kube-apiserver — the front door; the only thing that talks to etcd
  • etcd — the database; all cluster state lives here
  • scheduler — assigns pods to nodes
  • controller-manager — runs the built-in control loops

On every node

  • kubelet — starts & supervises containers
  • kube-proxy — wires up Service networking

Appendix: the API server is the hub

  • Everything is a REST resource at /apis/<group>/<version>/...
  • Components don't call each other — they watch the API.
  • A watch is a long-lived stream of add/update/delete events.
kubectl get pods --watch
# the same mechanism every controller uses

Appendix: how a pod "responds" to a resource

  1. A process in a pod opens a watch on some resource type.
  2. The API server streams every change to that process.
  3. The process reacts — creating, updating, deleting other resources.
  4. It writes results back to the API (e.g. a status).

That process is a controller. That's the whole trick.

Extending the API: CRDs

A CustomResourceDefinition teaches the API server a brand new resource type — with its own schema, validation, and kubectl support.

apiVersion: apiextensions.k8s.io/v1
kind: CustomResourceDefinition
metadata:
  name: reportrequests.reports.workshop.io
spec:
  group: reports.workshop.io
  names: { kind: ReportRequest, plural: reportrequests }
  scope: Namespaced
  versions: [ ... schema ... ]

A CRD alone does nothing

After applying a CRD you can do this…

kubectl apply -f my-reportrequest.yaml
kubectl get reportrequests

…but the object just sits there. It's only data.

You will feel this in step 03.

The operator / controller pattern

Pair a CRD (the data) with a controller (the behaviour):

CRD            →  a new noun in the API
Controller     →  the verb that makes it real
Together       →  an "Operator"

Encodes the knowledge a human operator would otherwise apply by hand.

Real-world use cases

  • Databases that provision & back themselves up (CloudNativePG, Zalando Postgres)
  • Certificate management (cert-manager: Certificate CRDs)
  • GitOps (Argo CD: Application CRDs)
  • Cloud resources from inside the cluster (Crossplane)
  • Your own domain objects — like today's ReportRequest.

Today's scenario: AI Report Queue

Users request a financial report. A controller turns each request into a real PDF — and the UI shows it happen live.

User → Web UI → creates ReportRequest (CRD)
                       │
                Controller sees it
                       │  creates a Job
                       ▼
                 Worker Job → calls mock-AI → renders PDF → MinIO
                       │
                Controller updates status → UI shows it live

Why this design teaches well

  • The controller creates other Kubernetes objects (Jobs) — the real operator pattern, visible in k9s.
  • Owner references mean deleting a request garbage-collects its Job.
  • The status subresource shows desired-vs-actual in action.
  • A few seconds of "work" make Pending → Processing → Completed visible.
